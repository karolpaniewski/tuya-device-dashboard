## Current System

`deviceTemperatureReadings` (SQLite via Drizzle, `src/server/db/schema.ts`)
is written every ~30s poll cycle by `src/server/workers/tuya-poller.ts`,
and read by the `device.temperatureHistory` tRPC query
(`src/server/api/routers/device.ts`), which buckets rows by range (1h/24h/
7d) for the History tab's chart on each device's detail modal.

**Tech stack:** Next.js 15, Drizzle ORM + libsql (SQLite), a persistent
polling worker process running alongside the Next.js server (not
serverless) — `tuya-poller.ts` already runs on a fixed tick interval, the
natural home for a periodic purge if one is added there, though that's an
implementation choice left open for planning.

**Users:** Facility manager / office administrator — same single-role
persona as the rest of the app, 2–5 person org. This change has no
user-facing surface; it's invisible infrastructure.

**Pain / gap:** No retention or purge job exists anywhere in `src/server/`
for this table. At 50 devices × 30s polling, that's ~144,000 rows/day,
growing unbounded. This was named as a risk in the roadmap when S-09
(temperature-history) was originally scoped, but never built — a
known-but-deferred follow-up cut for speed under S-09's after-hours
timeline, not an oversight.

**Must preserve:**
- The History tab's existing 1h/24h/7d range queries must keep returning
  the same data they do today for any range a purge policy doesn't touch.
- The 30s poll tick's existing cadence and reliability — a purge job must
  not slow down or risk breaking the live-reading path that feeds the
  dashboard's online/offline + current-temperature display.

## Vision & Problem Statement

**Change:** Add a periodic job that deletes `deviceTemperatureReadings`
rows older than a retention window, so the table stops growing without
bound. This is a pure infrastructure/maintenance addition — no new
user-facing capability, no change to what the History tab shows within
its retention window.

**Insight:** The growth risk was already identified and explicitly
accepted as a deferred follow-up when temperature-history shipped — this
isn't new information, it's closing a known gap before it becomes an
actual operational problem (disk usage, slower queries on an
ever-growing table, eventual restart/backup pain).

## User & Persona

**Role:** Facility manager / office administrator — same persona as the
rest of the app. No new persona; this change is invisible to them as
long as the History tab keeps working within its retention window.
**Pain moment:** N/A directly — no user currently experiences this as
pain. The "moment" this protects against is a future one: degraded query
performance or a runaway database file size if usage continues unchecked.

## Access Control

No changes planned — current model preserved: NextAuth email + password
login, single flat role, full access for the one effective user type.
This is a backend-only infrastructure job with no user-facing access
surface; auth and roles are untouched.

## Success Criteria

### Primary
The smallest end-to-end slice, proving the whole thing works:
1. A periodic job runs (on the existing poller's tick, or an equivalent
   scheduled trigger — left to planning) and deletes
   `deviceTemperatureReadings` rows older than 7 days.
2. Rows inside the 7-day window are never touched — verified by checking
   that the History tab's 1h/24h/7d ranges all still return exactly the
   data they did before the purge ran.
3. The job's execution (rows deleted, or zero rows deleted) is logged via
   the existing structured logger (matching the project's S-07
   observability convention), so a purge run is visible/auditable without
   needing a dedicated UI.

### Secondary
None identified — this is a narrow, single-purpose infra job with no
nice-to-have surface beyond the primary slice.

### Guardrails
- The job must never delete a row inside the 7-day retention window —
  this is the single most important property; the alternative (silent
  data loss, a chart that goes blank for a range that used to work) is
  worse than not building the job at all.
- The job must not contend with or slow down the 30s poll tick that
  feeds the dashboard's live online/offline + current-temperature
  display — it runs on its own cadence (e.g. once per day/hour, not
  every poll tick) regardless of where it's triggered from.
- The existing `device.temperatureHistory` query and its bucketing logic
  are otherwise unchanged — this job only deletes rows, it doesn't alter
  how surviving rows are read or bucketed.

**Timeline:** well under a week of after-hours work — a single delete
query, a trigger point, and a unit test for the retention-window boundary
math.

## User Stories

### US-01: Operator relies on the purge job to keep temperature history bounded without losing recent data

- **Given** `deviceTemperatureReadings` has accumulated readings older
  than 7 days alongside readings from the last 7 days
- **When** the periodic purge job runs
- **Then** rows older than 7 days are deleted, rows within the last 7
  days are untouched, and the run's outcome (rows deleted, or zero) is
  logged

#### Acceptance Criteria
- A device's History tab shows identical 1h/24h/7d chart data before and
  after a purge run, for any data point within the retention window
- A row exactly at the 7-day boundary is handled consistently (defined
  during planning: either always kept or always purged, not ambiguous)
- The purge job's log line appears regardless of whether any rows were
  actually deleted that run
- The 30s poll tick's reading-insert cadence shows no measurable
  slowdown or contention from the purge job running

## Business Logic

**No domain logic change.** This is an infrastructure/technical change —
a periodic delete query and nothing else. No new decision is made for
the user; the History tab's existing chart-range logic is unaffected for
any data still inside the retention window.

## Non-Functional Requirements

- A device's most recent 7 days of temperature readings are always
  retrievable via the History tab, regardless of when the purge job last
  ran.
- The purge job's execution does not introduce measurable latency or
  delay to the 30s poll tick's reading-insert path.
- The purge job's outcome is observable without inspecting the database
  directly — every run produces a log line, consistent with this
  project's existing structured-logging convention.

## Constraints & Preserved Behavior

- **No other consumer to preserve:** `device.temperatureHistory` is the
  only reader of `deviceTemperatureReadings` — no export, backup, or
  reporting integration depends on rows surviving past 7 days.
- **No data migration:** no schema change is required — the purge job
  operates on the existing table as-is. (If planning determines an index
  on the timestamp column would make the delete query efficient at
  scale, that's an additive, backward-compatible migration, not a
  breaking one.)
- **First-run backlog is acceptable as a single bulk delete:** the table
  currently has no purge history, so the first run will delete the
  entire backlog of rows older than 7 days in one pass — accepted as
  fine at this app's scale (small org, modest accumulated row count); no
  batching/throttling required.
- **No backward-compatibility risk:** no existing external API consumers
  touch this table directly.
- No special deployment window or CI/CD change — same lint+typecheck+test
  gate every other change in this project goes through.

## Non-Goals

- No change to product type (web app) or user base/scale (small,
  2–5 person org) — this is a backend-only infra job, invisible to users.
- No hard deadline; after-hours-only work, matching every other change in
  this project.
- Avoid: a configurable retention window — 7 days is hardcoded, not
  exposed as an env var or Settings UI control. Changing it later is a
  code change, not a config change; making it configurable now would be
  speculative scope for a need that doesn't exist yet.

## Forward: tech-stack

The following implementation-shaped decisions came up during shaping but
are explicitly NOT decided here — they belong to downstream planning:

- **Trigger mechanism:** whether the purge runs on the existing
  `tuya-poller.ts` tick (gated to fire roughly daily rather than every
  30s tick), piggybacks on `automation-scheduler.ts`'s existing tick
  loop, or gets its own dedicated interval/cron-like trigger. All three
  are consistent with this app's existing "persistent worker process,
  not serverless" architecture (per Current System) — left to planning.
- **Boundary semantics:** whether the 7-day cutoff is inclusive or
  exclusive at the exact boundary (a row exactly 7 days old) — a small
  detail, but one that should be decided explicitly and tested, not left
  ambiguous (see US-01's acceptance criteria).
- **Whether to add an index** on the timestamp column to make the
  delete query efficient — an additive, backward-compatible migration if
  planning determines it's needed at this app's row-count scale.
