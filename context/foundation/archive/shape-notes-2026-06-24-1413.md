> Seed idea: Interactive 2D floor plan ("digital twin") — instead of showing
> devices only in lists/tables, let the user place them spatially. User
> uploads a floor-plan file (or draws simple room rectangles on a CSS grid),
> then drag-and-drops sensor/valve icons onto the map. Devices pulse
> red/blue by temperature status; clicking a device icon opens the existing
> valve control panel. Explicitly framed by the user as also demonstrating
> technical depth (absolute positioning, SVG manipulation, libraries like
> react-konva).

## Current System

Dashboard shows devices grouped by room as cards in a vertical list/grid
(`room-group.tsx`), each card showing temperature, online status, setpoint,
and (since S-05/S-10) a comfort-threshold badge and alert-sent indicator.
There is no spatial/visual representation of the physical space today —
rooms are named sections in a list, with no indication of how rooms relate
to each other physically.

Stack: Next.js 15, tRPC v11, Drizzle ORM + libsql (SQLite), Tuya LAN
polling/control. No SVG/canvas rendering library currently in the stack.

Users: facility manager / office administrator, 2–5 person org, flat
single-role access model (NextAuth email + password, no roles) — same
persona as the rest of the app.

**Pain / gap** (in the user's own words):
- **No spatial context:** Lists/tables, even sorted, don't show
  relationships between rooms. If "Room 4" is cold, the table alone doesn't
  tell the user it's a corner room on the north wall that always cools
  down — a 2D floor plan immediately supplies that context.
- **Naming/onboarding problem:** Users (especially in offices, or new
  household members) rarely remember which valve is
  "Grzejnik_Korytarz_Północ_2" — but they know where they physically
  stand. A 2D floor plan removes the need to decode device names into
  physical location.
- **Glanceability with many rooms:** With 10–15+ rooms, scanning a list for
  warning badges (S-05) becomes tedious. A floor plan with red/blue
  pulsing points lets the admin spot the problem area across an entire
  floor in one glance.

The user explicitly named this change's primary driver as a
portfolio/skill-demonstration feature (absolute positioning, SVG/canvas
manipulation, drag-and-drop) rather than a pre-existing, deeply-felt user
complaint — recorded as such, not dressed up as discovered user pain.

**Must preserve** (in the user's own words):
- **Core control loop criticality:** The app's primary purpose is
  temperature management (S-04, S-20). If the floor-plan script fails
  (rendering-library bug, unsupported floor-plan file), the user must
  still be able to turn heating on/off. A UI failure in the visualization
  layer must never block the critical business path.
- **Mobile constraints (S-08):** A 2D floor plan is inherently hard to use
  on a 375px viewport (would need pinch-to-zoom and panning). The existing
  tile/list view is the natural, optimal mobile solution. The floor plan
  may be hidden by default on mobile, or available only in landscape.
- **Accessibility (a11y):** Screen readers cannot interpret a visual
  Canvas/SVG map. The existing tiles/tables, built on semantic HTML, are
  what keeps the dashboard accessible.
- **Bulk operations:** The floor plan suits point intervention (click one
  device). For sorting all devices by a property, or turning off heat in
  five rooms at once, the existing table/list view remains the better
  tool — the floor plan does not replace it.

## Vision & Problem Statement

**Change:** Add an interactive 2D floor-plan view ("digital twin") as an
additional visualization layer, alongside the existing room-card list —
not a replacement for it. The admin uploads a floor-plan image or draws
simple room rectangles on a grid, then drags sensor/valve device icons
onto their physical locations on the map. Placed devices visually pulse
according to temperature/threshold status; clicking a device icon on the
map opens the same valve control panel the existing list view already
uses.

**Insight:** This is primarily a technical-depth showcase (working with
absolute positioning, SVG/canvas manipulation, and drag-and-drop
libraries such as react-konva) rather than a response to an existing,
deeply-felt user complaint. The underlying spatial-context and
naming-recall pains are real and named above, but the user is explicit
that proving this kind of UI/rendering capability is the primary driver
for building it now.

## User & Persona

**Role:** Facility manager / office administrator — same persona as the
rest of the app, 2–5 person org. No new persona is introduced by this
change.
**Device:** Primarily desktop/wide-viewport — the floor plan itself is
explicitly not optimized for the 375px mobile case (see Must preserve);
the existing list view continues to serve mobile.
**Pain moment:** Looking at a multi-room list and needing to mentally
translate device names or row positions into "where in the office is
this", especially when onboarding a new team member who doesn't yet
know which valve name maps to which physical room, or when scanning many
rooms for which one needs attention.

## Access Control

No changes planned — current model preserved: NextAuth email + password
login, single flat role, full access for the one effective user type. The
floor plan is visible and editable (uploading the plan, placing devices)
by the same single role that already has full dashboard access. No new
accounts or roles are introduced by this change.

## Success Criteria

### Primary
The smallest end-to-end slice, proving the whole stack works together:
1. Admin uploads a static floor-plan image (PNG/JPG) via Settings.
2. Admin opens a new "Map View"; the floor plan renders, with a list/drawer
   of devices not yet placed on the map.
3. Admin drags a device icon from that list onto its physical location on
   the floor plan; the position is saved.
4. The placed device renders on the map and visually changes color/glow
   based on live temperature/threshold status (reusing the existing Tuya
   polling data — no new telemetry path).
5. Admin clicks the device on the map → opens the existing device
   management modal (S-16) → adjusts setpoint → closes modal. Same control
   path the list view already uses.

No custom room-drawing tool (no wall-dragging, no grid-snapping) in this
MVP — the floor plan is a static uploaded image; device positions are
absolute, percentage-based coordinates over that image.

### Secondary
None for this MVP — the shared layout animation idea (clicking a device
node morphs into the device modal) was considered and dropped during the
Socrates round on FR-012: the implementation complexity wasn't judged
worth it for a nice-to-have. The primary flow is the whole scope.

### Guardrails
- The core control loop (turning heat on/off, adjusting setpoints) must
  remain fully usable via the existing list/table view even if the
  floor-plan rendering fails (unsupported file, rendering-library bug) —
  a UI failure in the map view must never block critical device control.
- Mobile/375px usability (S-08) is unaffected — the existing list/table
  view continues to serve mobile; the floor plan is not required to work
  on narrow viewports.
- Accessibility is unaffected — the existing semantic-HTML list/table
  view remains the accessible path; the floor plan is a visual-only
  addition, not a replacement.
- Bulk operations (multi-room actions, sorting by a property) remain
  available via the existing list/table view — the floor plan suits
  single-device, point-in-time interaction, not bulk actions.

**Timeline:** roughly three weeks of after-hours work or less — file
upload + storage, a new Map View page, drag-and-drop placement,
percentage-coordinate persistence, live status-color rendering, and
wiring the existing device modal to map clicks.

## Forward: technical-roadmap

The following ideas came up during shaping but are explicitly deferred
beyond this MVP — captured here so they aren't lost, but they are NOT
part of this change's scope (see also Non-Goals below):

- **Thermal heatmap overlay** — radial, semi-transparent gradients per
  device that merge into a literal floor heatmap when multiple heaters
  are active.
- **SVG polygon room mapping** — parse an uploaded `.svg` floor plan so
  the app understands actual room boundaries; hover/click highlights the
  whole room shape, not just a point.
- **Pan & zoom with momentum scrolling** — a gesture-control wrapper for
  navigating large floor plans, particularly on mobile.
- **Snap-to-grid placement** — invisible grid constraint during
  drag-and-drop so device icons align neatly.
- **Self-hosted asset pipeline** — serve floor-plan images via a local
  storage + reverse-proxy pipeline instead of a cloud object store, for
  on-premise file handling.
- **Real-time multi-user cursor tracking** — broadcast other admins'
  cursor positions / live edits on the map via WebSockets.
- **Spatial automation hooks** — rules that reason about device
  *physical* proximity (e.g. "boost the heater opposite a drafty window
  sensor"), once the system has placement data to reason over.
- **Read-only mobile floor-plan view** — a non-interactive (no
  drag-and-drop) view of the map on narrow viewports, instead of hiding
  it entirely. Deferred from this MVP (FR-010) to keep scope tight.
- **Shared layout animation into the device modal** — clicking a map
  device node morphs smoothly into the S-16 modal instead of it
  instantly appearing. Considered as the MVP's Secondary criterion,
  dropped during the Socrates round (FR-012) as not worth the added
  complexity for a nice-to-have.

Implementation-level notes from shaping (also downstream of this PRD, not
part of it): a `map_x_pct` / `map_y_pct` (float) pair on the relevant
device-placement record, a `floor_plan_image_url` on the site/room record,
a drag-and-drop library such as `@dnd-kit/core` (already a project
dependency-adjacent pattern — `@dnd-kit/sortable` is already used
elsewhere in this app) or `react-use-gesture`, and a small percentage-
coordinate utility for translating a drop event into relative map
coordinates.

## User Stories

### US-01: Admin places a device on the floor plan and controls it spatially

- **Given** an admin has uploaded a floor-plan image for their site and
  has at least one unplaced device
- **When** they drag a device icon onto the floor plan, and later see it
  change color as its temperature crosses a threshold
- **Then** they can click the device node to open the existing setpoint
  control modal and adjust it, exactly as they would from the list view

#### Acceptance Criteria
- A device's map position persists across page reloads
- A floor-plan rendering failure does not prevent setpoint changes via
  the existing list view
- The map view is never required to control a device — the list view
  remains a complete, independent control path
- A placed device can be re-positioned or returned to the unplaced
  roster after initial placement

## Business Logic

**No domain logic change.** The floor plan is a visualization layer: it
renders the existing S-05 threshold decision (OK / Too Cold / Too Hot)
spatially, as a colored map node instead of a badge — it does not decide
anything new for the user. The input (a room/device's current badge
state) and the output (the same three-state classification) are
unchanged from S-05; only the presentation surface is new.

## Non-Functional Requirements

- An uploaded floor-plan image renders on the Map View without a
  noticeably different load time from the rest of the dashboard.
- A device's map position update (drag-and-drop) is reflected without
  requiring a manual page reload.
- Existing externally-observable behaviors (dashboard load time, badge
  accuracy, valve control responsiveness, mobile usability) do not
  regress — this is a pure addition, not a modification of any existing
  user-facing guarantee.

## Constraints & Preserved Behavior

- **No data migration:** this is purely additive — new optional fields
  (a floor-plan image reference per site, a map x/y position per device)
  with no backfill and no reshaping of any existing table. A device with
  no map position simply appears in the unplaced roster.
- **No backward-compatibility concerns:** there are no existing external
  API consumers, exports, or integrations touching device/site/room data
  today, so nothing downstream can break from these additive fields
  (same conclusion as the external-notifications change).
- Must respect the existing site-scoping convention: the Map View only
  shows devices belonging to the currently active site (FR-007).
- Must not introduce a new telemetry/polling path — live device status
  on the map reuses the existing Tuya polling data already powering the
  list view.
- The existing list/table view, valve control path, mobile experience
  (S-08), and accessibility posture must all continue working exactly as
  today, independent of the floor plan's state (see Current System →
  Must preserve, and the Guardrails in Success Criteria).

## Non-Goals

- No change — existing product type (web app) and existing user base
  (small office/admin, 2–5 person org); this feature doesn't alter
  either.
- No hard deadline; after-hours-only work, no special CI/CD or
  deployment-window constraints beyond the existing lint+typecheck+test
  gate (S-06).
- Avoid: custom room-drawing/wall-snapping tool — the floor plan is a
  static uploaded image; no drawing rooms, no wall-dragging, no
  grid-snapping (already stated under Success Criteria → Primary).
- Avoid: SVG polygon room-boundary parsing — devices are point positions
  on an image; no understanding of actual room shapes/boundaries in this
  MVP.
- Avoid: pan/zoom/gesture controls and full mobile map support — the map
  is desktop/wide-viewport only for this MVP; the existing list view
  continues to serve mobile (375px, S-08).
- Avoid: real-time multi-user collaboration — no live cursor tracking or
  broadcasting edits between simultaneous admins.
- Avoid: spatial automation rules — no rules that reason about device
  physical proximity (e.g. "boost the heater opposite a drafty window").
- Avoid: thermal heatmap overlay — single-device color status only, no
  merged radial-gradient heatmap visualization.
