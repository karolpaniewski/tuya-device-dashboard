## Current System

Tuya Device Dashboard to web app (Next.js 15 + shadcn/ui + tRPC + libsql/SQLite) dla facility managerów (1–5 osób per site). Dashboard pokazuje karty pokojów z temperaturą i statusem urządzeń. Kliknięcie room card otwiera centered Dialog (`RoomModal`) który pokazuje listę urządzeń pokoju i automation modes które go targetują. Ten sam `RoomModal` jest używany w automation-flow canvas (klik room-node). Temperatura historyczna jest już dostępna przez `api.device.temperatureHistory` i renderowana w `room-temperature-panel.tsx` (Recharts `LineChart`).

## Vision & Problem Statement

Facility manager przegląda dashboard z dziesiątkami kart pokojów. Kiedy kliknie pokój żeby sprawdzić stan urządzeń, centered Dialog (`RoomModal`) blokuje cały widok floty — nie może równolegle śledzić innych pokojów. Zamknięcie i ponowne otwarcie następnego pokoju to zbędne kliknięcia. Slide-in panel z prawej strony pozwoli przeglądać stan pokoju bez utraty kontekstu dashboardu, a historia temperatury w panelu da pełny wgląd bez wchodzenia do osobnego widoku urządzenia.

## Scope of Change

- FR-001: Facility manager otwiera room quick-overview panel klikając room card na dashboardzie. Priority: must-have. Change: modified (zastępuje centered Dialog).
  > Socrates: Kontr-argument rozważony: "click handler może kolidować z istniejącą nawigacją." Rozwiązanie: kept — room card nie ma nawigacji per se, otwierała RoomModal; zmiana zastępuje ten modal Sheetem.
- FR-002: Facility manager widzi aktualną temperaturę pokoju i 24h trend w nagłówku panelu. Priority: must-have. Change: new.
- FR-003: Facility manager widzi 24h historyczny chart temperatury w panelu. Priority: must-have. Change: new (wcześniej dostępne tylko w device modal).
  > Socrates: Kontr-argument rozważony: "pokój bez sensora nie ma danych — chart będzie pusty." Rozwiązanie: sekcja historii temperatury jest ukryta gdy pokój nie ma żadnego sensora (empty state, nie placeholder).
- FR-004: Facility manager widzi listę urządzeń pokoju z inline statusem (online/offline, setpoint, valve). Priority: must-have. Change: preserved (ta sama treść co RoomModal).
- FR-005: Facility manager widzi listę aktywnych trybów targetujących pokój. Priority: must-have. Change: preserved.
- FR-006: Facility manager może włączyć/wyłączyć heat dla pokoju bezpośrednio z panelu. Priority: must-have. Change: new (heat toggle dostępny wcześniej tylko w device-overview).
  > Socrates: Kontr-argument rozważony: "skąd panel zna aktualny heat state?" Rozwiązanie: device.overview query jest już załadowany przez dashboard — panel czyta heat state stamtąd; nie wymaga nowego endpointu.
- FR-007: Facility manager zamyka panel przez X, Esc lub klik overlay. Priority: must-have. Change: new behavior.
- FR-008: RoomModal w automation-flow canvas pozostaje bez zmian. Priority: must-have. Change: preserved.
  > Socrates: Kontr-argument rozważony: "spójność UX wymaga tego samego Sheetu w canvas." Rozwiązanie: kept jako preserved — canvas ma swoją logikę (modalRoomId state, ograniczona przestrzeń); zmiana poza scopem tego featurea.

## Non-Goals

- Nie edytuje urządzeń — panel jest read-only poza heat toggle; setpoint, schedules i device settings pozostają w device modal.
- Nie zastępuje RoomModal w automation-flow canvas — canvas zachowuje centered Dialog.
- Nie persystuje stanu panelu — zamknięcie i ponowne otwarcie nie pamięta scroll ani wybranego pokoju.
- Nie targetuje mobile — dashboard nie jest mobile-first surface; Sheet może być użyteczny na mobile, ale nie jest testowaną powierzchnią dla tego featurea.

## Business Logic Changes

No domain logic change. This is an infrastructure/UX change — panel prezentuje te same dane (urządzenia, tryby, temperatura) przez inny affordance (Sheet zamiast centered Dialog), i eksponuje istniejącą operację (api.room.toggleHeat) w nowym miejscu.

## Non-Functional Requirements

- Panel otwiera się w < 100 ms (animacja wejścia Sheet); treść dostępna od razu jeśli device.overview jest w cache tRPC.
- Historia temperatury może wyświetlać skeleton podczas ładowania (api.device.temperatureHistory), ale reszta panelu nie czeka na chart.

## Constraints & Preserved Behavior

- `tuya-automation-flow.tsx` nie jest modyfikowany — `RoomModal` pozostaje jedynym popup w canvas.
- `api.room.toggleHeat` mutation zachowuje istniejące zachowanie (toast success/error, invalidation device.overview).
- `device.overview` query nie jest duplikowane — panel konsumuje ten sam cache co dashboard.
- shadcn/ui Sheet komponent wymaga instalacji (`npx shadcn@latest add sheet`); nie zmienia żadnych istniejących komponentów UI.

## User Stories

### US-01: Room Quick-Overview Panel

- **Given** zalogowany facility manager na dashboardzie z listą pokojów
- **When** kliknie dowolną room card
- **Then** slide-in Sheet otwiera się z prawej strony; dashboard pozostaje widoczny i nie jest blokowany przez overlay pełnoekranowy

#### Acceptance Criteria
- Panel zawiera: temperaturę aktualną + 24h trend chip; 24h LineChart historii temperatury; listę urządzeń z statusem; listę trybów; przycisk heat toggle
- Kliknięcie X, Esc lub overlay poza panelem zamyka panel
- RoomModal w automation-flow canvas nie zmienia zachowania

## Success Criteria

### Primary
- Kliknięcie room card na dashboardzie otwiera slide-in Sheet z prawej strony, dashboard pozostaje widoczny i przewijalny w tle; panel pokazuje temperaturę aktualną, 24h chart, urządzenia, tryby i heat toggle.

### Secondary
- Panel ładuje dane natychmiast (cache tRPC), bez osobnego stanu ładowania widocznego użytkownikowi przy powtórnym otwarciu tego samego pokoju.

### Guardrails
- RoomModal w automation-flow canvas działa bez zmian — klik room-node nadal otwiera centered Dialog.
- Heat toggle w panelu zachowuje się identycznie jak heat toggle w device-overview (ten sam mutation, te same toast notifications).

## Access Control

No access control changes — current model preserved. Panel jest widoczny dla tych samych zalogowanych użytkowników co obecna dashboard. System używa email + hasło auth (rola admin). Żadnych nowych ról ani poziomów dostępu.

## User & Persona

**Facility manager** — zarządza 5–50 pokojami w jednym budynku. Sprawdza dashboard kilka razy dziennie żeby zweryfikować stan grzewczy. Potrzebuje szybkiego wglądu w konkretny pokój bez odrywania się od widoku floty. Frustruje go konieczność zamykania i otwierania kolejnych modali żeby porównać pokoje.
