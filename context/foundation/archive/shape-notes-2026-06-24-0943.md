## Current System

Dashboard shows per-room threshold badges (S-05: OK / Too Cold / Too Hot),
computed from the current temperature reading against a configured
per-room threshold. The badge is only visible if the admin is actively
looking at the dashboard — there is no outbound signal of any kind today.

Stack: Next.js 15, tRPC v11, Drizzle ORM + libsql (SQLite), Tuya LAN
polling/control — unchanged, no new stack needed for the existing parts of
the system this change builds on.

Users: facility manager / office administrator, 2–5 person org, flat
single-role access model (NextAuth email + password, no roles).

**Pain / gap:**
- A room crossing its threshold (too cold → heating must be turned on; too
  hot → it must be turned off) is invisible unless the admin happens to be
  looking at the dashboard at that moment. Today they find out late — e.g.
  the next morning, or after returning to the office — by which point the
  room may have been out of range for hours.

**Must preserve:**
- Threshold computation and badge logic (S-05) is unchanged — this is a
  new notification layer on top of the existing mechanism, not a redesign
  of how thresholds are evaluated.
- Notifications must never block or slow down valve control. The alerting
  path is strictly read-only with respect to device control — it observes
  threshold state and sends a message; it never sits in the call path of
  `sendValveStateCommand` or any other device command.

## Vision & Problem Statement

**Change:** Add an outbound notification (email / SMS / push) sent when a
room's threshold badge transitions into a violated state (Too Cold / Too
Hot), so the admin team learns about it without needing to be watching the
dashboard. This is a new integration — the app has never sent anything to
an external channel before.

**Insight:** Two things make this non-obvious, not just "add an API call":
1. There is no existing outbound infrastructure in the app at all (no
   email/SMS/push provider, no API key, no delivery mechanism) — this is
   the first integration of this kind.
2. A room's temperature can oscillate around a threshold repeatedly in a
   short window; without throttling/debounce, that would flood the team
   with duplicate alerts for the same ongoing situation instead of one
   alert per violation episode.

## User & Persona

**Role:** Facility manager / office administrator team (2–5 person org) —
same persona as the rest of the app. Unlike the rest of the app's
single-effective-user model, notifications go to **multiple recipients**
(the whole team, not just one admin) — this is the first place in the
product where "who receives this" is a list rather than implicitly "the
one logged-in admin."
**Device:** Recipients receive alerts on their own phone/email client —
outside the dashboard's desktop/mobile-web surface entirely.
**Pain moment:** A room drifts out of its comfort range while no one is
looking at the dashboard (after hours, weekend, away from desk) — the team
only finds out once someone happens to check, by which point the room may
have been out of range for a long time.

## Access Control

No changes planned — current model preserved: NextAuth email + password
login, single flat role, full access for the one effective user type. No
new accounts or roles are introduced by this change.

Notification recipients are **not** modeled as accounts. They are a
static list of contacts (email addresses / phone numbers) maintained in
application settings by the admin — a plain contact list, not a new
identity or role boundary. The same global list applies to every
threshold violation; per-room or per-threshold recipient targeting is not
part of this change (see Non-Goals).

## Success Criteria

### Primary
A room's threshold badge transitions into a violated state (Too Cold /
Too Hot) → the system sends one email to the configured contact list
stating which room and which threshold was violated. Repeated oscillation
around the threshold in a short window produces one alert per violation
episode, not one per fluctuation.

### Secondary
The email includes a link that opens the dashboard directly to the
affected room, so a recipient can act in one click instead of navigating
to find it.

### Guardrails
- Notification sending never blocks, delays, or sits in the call path of
  valve control (`sendValveStateCommand` or any other device command) —
  the alerting path is strictly observational.
- Threshold computation and badge logic (S-05) is unchanged by this
  change — notifications are read-only consumers of existing badge state.
- No SMS/push in this MVP — explicitly deferred (see Non-Goals).

**Timeline:** roughly one week of after-hours work — email integration,
threshold-violation detection hook, throttle/debounce for oscillation, and
a settings screen for the contact list.

  > Socrates: Counter-argument considered: should S-05 stay completely

## Business Logic

**A room entering a threshold-violation state fires exactly one alert per
violation episode — no further alert is sent for the same ongoing
violation, only when the room returns to OK and violates again.**

This is a new rule, not a modification of the existing threshold/badge
rule (S-05). The input is the room's badge state over time (OK / Too Cold
/ Too Hot) as it already exists; the rule's job is to decide, from that
state history, which transitions deserve a notification and which don't.
The output is a yes/no decision per transition: a transition into a
violated state from OK fires an alert; staying in a violated state, or
fluctuating within it, does not.

The user encounters this as: they get exactly one email per real-world
episode of a room going out of range, never a flood of repeated emails
for the same ongoing problem — and conversely, if the same room drifts
out of range again after having recovered, that's treated as a new,
distinct episode worth a fresh alert.

## User Stories

### US-01: Room threshold violation triggers an email alert

- **Given** a room has a configured threshold (S-05) and at least one
  contact is configured in settings
- **When** the room's badge transitions from OK into a violated state
  (Too Cold or Too Hot)
- **Then** the system sends one email to the configured contact list
  naming the room and the violated threshold, with a link to the room on
  the dashboard, without delaying or blocking any valve control command;
  no further alert is sent for the same episode until the room returns
  to OK and violates again

#### Acceptance Criteria
- A room with no configured threshold never produces an alert
- Exactly one email is sent per violation episode, regardless of how many
  times the temperature fluctuates around the threshold within that
  episode
- The dashboard badge shows an "alert sent" indicator for a room with an
  active, already-notified violation
- Email sending is asynchronous and never appears in the call path of
  `sendValveStateCommand` or any other device command
- An invalid email format is rejected at the point the admin saves the
  contact, not silently dropped at send time

## Non-Functional Requirements

- An alert is delivered within roughly 1 minute of the threshold
  violation being detected.
- Configured contact addresses (email/phone) are never written to
  application logs in plaintext or otherwise — no leak of contact PII
  through logging.
- Existing externally-observable behaviors (dashboard load time, badge
  accuracy, valve control responsiveness) do not regress — this is a
  pure addition, not a modification of any existing user-facing
  guarantee.

## Constraints & Preserved Behavior

- **Network model clarification:** the product's "LAN-only, no internet"
  guarantee applies specifically to Tuya device control — Tuya's cloud
  portal is unavailable in this network and device communication stays
  on the LAN. The application **server** itself has normal outbound
  internet access; this was never a full network-isolation requirement.
  Sending email via an external SMTP/email provider does not violate the
  product's existing network constraint.
- No data migration: this is purely additive — a new contacts table and
  new alert-episode tracking, no existing table's shape changes other
  than the small "alert sent" indicator on the room/badge read path
  (FR-005).
- No backward-compatibility concerns: there are no existing external API
  consumers or integrations touching room threshold state today, so
  nothing downstream of S-05 can break from this addition.
- Must respect the existing valve-control call path: no awaited call to
  the email provider — or to anything else outside the polling/control
  loop — may sit between a threshold read and a valve command.

## Non-Goals

- No change — existing product type (web app) and existing user base
  (small office/admin, 2–5 person org); the contact list is multiple
  recipients, but the organization scale itself doesn't change.
- No hard deadline; budget is the already-recorded ~1-week after-hours
  estimate.
- Avoid: SMS/push notifications in this MVP — email only; SMS/push are
  explicitly deferred to a later iteration.
- Avoid: recipient targeting per room/threshold — one global contact
  list applies to every alert, not per-room or per-threshold recipients.
- Avoid: alert history/log UI — no screen listing past notifications; the
  badge's "alert sent" indicator and the email itself are the only
  signals in this MVP.
- Avoid: re-alert / reminder for an ongoing violation — one alert per
  episode, no periodic reminder while a room stays out of range.
