## Current System

Two places control a device's setpoint today: (1) the dashboard device
card has a compact +/− button pair (0.5°C steps, no slider, no dial —
`device-card.tsx`'s `SetpointControl`), (2) the device detail modal
(`device-modal.tsx`, opened via a plain Base UI dialog) has a linear
`<input type="range">` slider. Clicking a device card opens that modal
instantly — no transition/morph animation; it's a standard dialog mount,
not a shared-layout expansion.

**Tech stack:** Next.js 15, React, Tailwind v4, Base UI (dialogs/popovers),
Recharts (charts). No animation library installed — no Framer Motion, no
View Transitions API usage anywhere yet.

**Users:** Facility manager / office administrator — same single-role
persona as the rest of the app, 2–5 person org.

**Pain / gap** (in the user's own words): the current controls work fine
functionally, but look and feel like a generic admin tool rather than a
polished consumer product — this is about perceived quality/craft, not
usability friction. "A premium product is recognized by how the app
flows."

**Must preserve:** setpoint accuracy & responsiveness — the new dial must
send the exact same setpoint mutation, with the same 5–35°C clamp and
0.5°C step, and visual polish must not introduce input lag or wrong
values reaching the valve.

## Vision & Problem Statement

**Change:** Replace two existing interaction points with richer,
physically-inspired ones: (1) the device card's +/− setpoint buttons
become an interactive circular thermostat dial (SVG/Canvas-based,
touch/mouse-draggable like a Nest device) with a background color that
shifts smoothly from cool blue to warm orange as it's turned; (2)
clicking a device card no longer opens an instant dialog — the card
visually expands ("shared layout" morph) into the modal's size and
position, then plays in reverse on close.

**Insight:** This is the first feature in the app that requires an
animation/interaction library (Framer Motion or the native View
Transitions API) — today's stack has none. The functional features (S-04
setpoint control, S-16 device modal) shipped first; this is the first
dedicated pass at visual/interaction quality, explicitly framed as a
premium-feel polish pass rather than a response to a usability complaint.

## User & Persona

**Role:** Facility manager / office administrator — same persona as the
rest of the app, 2–5 person org. No new persona is introduced by this
change.
**Pain moment:** Adjusting a device's setpoint or opening its detail view
and experiencing it as functionally adequate but visually/interactionally
unremarkable — "doesn't feel like a premium product."

## Access Control

No changes planned — current model preserved: NextAuth email + password
login, single flat role, full access for the one effective user type.
This is a pure UI/interaction feature; auth and roles are untouched.

## Success Criteria

### Primary
The smallest end-to-end slice, proving the whole stack works together:
1. User opens the dashboard.
2. User clicks a device card.
3. The card visually morphs/expands (shared-layout animation) into the
   modal at its full size and position — not an instant dialog mount.
4. For valve-type devices, the modal's setpoint control is now an
   interactive circular thermostat dial (SVG/Canvas, drag-to-rotate via
   touch or mouse), with a background color that shifts smoothly from
   cool blue to warm orange as it's turned.
5. The dial sends the exact same `device.setpoint` mutation as today,
   with the same 5–35°C clamp and 0.5°C step.
6. Closing the modal reverses the morph back into the card.

Sensor/plug device cards (no setpoint) still use the shared-layout morph
on open/close; they keep their existing (dial-less) modal content
unchanged.

### Secondary
Haptic/sound feedback while turning the dial — a subtle click/tick as it
passes each 0.5°C increment. Nice-to-have; not required for the MVP to
count as working.

### Guardrails
- The dial must send the exact same setpoint mutation as the current
  +/− buttons, with the same 5–35°C clamp and 0.5°C step — visual polish
  must not introduce input lag or an incorrect value reaching the valve.
- Sensor/plug device cards (no setpoint) are unaffected by the dial work
  and continue to open via the shared-layout morph with their existing
  modal content (History tab, temperature chart, Automations tab)
  unchanged.
- The modal's existing tabs (Overview/History/Automations) and the
  temperature chart continue to render and function exactly as today —
  the morph animation wraps the modal's entrance/exit, it does not alter
  its internals.

**Timeline:** roughly three weeks of after-hours work or less — an SVG/
Canvas thermostat dial component with drag gesture handling, a
shared-layout animation library integration (Framer Motion or View
Transitions API) wrapping the existing card→modal open/close, and wiring

## Functional Requirements

### Thermostat dial

- FR-001: User can drag/rotate a circular thermostat dial (SVG/Canvas)
  with touch or mouse to adjust a valve device's setpoint, on both the
  dashboard device card (compact size) and the device detail modal
  (larger size). The card's dial remains fully interactive (not just a
  visual indicator) but is designed with a generous touch target /
  drag deadzone so it stays usable despite its compact size; the
  modal's larger dial is the primary, most precise control surface.
  Priority: must-have. Change: new.
  > Socrates: Counter-argument considered: a drag-to-rotate gesture on a
  > small card-sized dial is error-prone/fiddly, undermining the
  > "premium feel" goal. Resolution: keep the card dial interactive,
  > but require a generous touch target / drag deadzone in its design;
  > the modal dial remains the primary precise control. FR revised to
  > state this explicitly.
- FR-002: The dial's background fill shifts smoothly from cool blue to
  warm orange as the setpoint value increases. The gradient is
  contained to the dial's own circular area, visually distinct from the
  separate room-status badge pill shown elsewhere on the card/dashboard.
  Priority: must-have. Change: new.
  > Socrates: Counter-argument considered: blue→orange could clash with
  > the existing Too-Cold/Too-Hot badge color language used elsewhere
  > in the app. Resolution: accepted the risk — the gradient lives
  > inside the dial's own contained shape, distinct in context/shape
  > from the badge pill, so confusion in practice is unlikely. FR
  > revised to note the containment explicitly.
- FR-003: The dial sends the same `device.setpoint` mutation as today,
  respecting the same 5–35°C clamp and 0.5°C step. Priority: must-have.
  Change: modified (replaces the card's +/− buttons and the modal's
  linear slider).
  > Socrates: Counter-argument considered: this might be trivially
  > obvious and not worth a dedicated FR, since it's just "don't break
  > the existing contract." Resolution: kept explicit anyway — this
  > project's convention (see floor-plan-map-view change) is to state
  > preserved contracts as defensive guardrail-FRs even when "obviously"
  > true, so it's testable in planning rather than assumed.
- FR-004: Sensor and plug device types (no setpoint) are unaffected — no
  dial appears on their cards or modal. Priority: must-have. Change:
  preserved.
  > Socrates: Counter-argument considered: since the dial is only ever
  > built for valve devices, this might be redundant/trivially true by
  > construction. Resolution: kept as an explicit guardrail FR anyway,
  > matching this project's established convention for preserved
  > behavior.

### Shared-layout transitions

- FR-005: Clicking a device card morphs/expands it (shared-layout
  animation) into the modal's size and position, instead of an instant
  dialog mount. The animation morphs the card's outer bounding box
  (position/size) consistently across all card types; each card's
  internal content (dial vs no dial, different readouts) cross-fades
  in/out rather than attempting to literally morph mismatched internal
  layouts between card types. Priority: must-have. Change: modified.
  > Socrates: Counter-argument considered: sensor, valve, and plug cards
  > have different internal content/heights — a shared-layout morph
  > could look smooth for one card type and visually broken for
  > another. Resolution: scope the morph to the card's outer bounding
  > box only; internal content cross-fades rather than reflows. FR
  > revised to state this explicitly.
- FR-006: Closing the modal reverses the morph animation back into the
  originating card, with its duration explicitly capped (e.g. <300ms)
  so it never feels sluggish to a user who wants to close quickly. The
  modal's existing tabs (Overview/History/Automations) and temperature
  chart continue to render and function exactly as today — the morph
  wraps entrance/exit only and does not alter modal internals. Priority:
  must-have. Change: modified.
  > Socrates: Counter-argument considered: (1) if the reverse-morph
  > duration is too long, users who just want to dismiss the modal
  > quickly will feel friction instead of polish. Resolution: cap the
  > animation duration explicitly. (2) A separate FR-007 stating modal
  > internals are preserved was judged redundant with this FR once
  > "wraps entrance/exit only" is stated here — merged into this FR
  > rather than kept as a near-duplicate line.

### Secondary

- FR-008: While dragging the dial, a subtle haptic tick (vibration API,
  where supported) fires at each 0.5°C increment crossed. No audio —
  haptic feedback is private to the person touching the device and
  never disturbs others nearby. Priority: nice-to-have. Change: new.
  > Socrates: Counter-argument considered: sound effects in a shared
  > office environment could be seen as unprofessional or annoying.
  > Resolution: dropped the audio option entirely; kept haptic-only
  > feedback, which doesn't have the shared-office downside. FR revised
  > to state this explicitly.

## Business Logic

**No domain logic change.** This is a pure visualization/interaction
change: the `device.setpoint` mutation, valve control behavior, and the
existing threshold/badge decision logic are all unchanged. Nothing new
is decided for the user — the dial and the morph animation only change
how an existing decision (the setpoint value) is captured and how an
existing view (the device modal) is presented.

## Non-Functional Requirements

- The morph animation and dial color transitions respect the user's
  reduced-motion preference (OS/browser level) — when set, the modal
  opens/closes instantly instead of morphing, and the dial's color
  transitions are immediate rather than animated.
- Dragging the dial updates its visual position and color immediately,
  with no perceptible lag between gesture and feedback, even while the
  underlying setpoint mutation is still in flight.
- Existing externally-observable behaviors — dashboard load time,
  setpoint accuracy, modal content correctness, and mobile usability of
  the rest of the dashboard — do not regress; this is a pure
  presentation-layer addition over an unchanged control contract.

## Constraints & Preserved Behavior

- **No data migration:** this change touches no backend or data model —
  no new columns, no new API endpoints, no persisted user preference.
- **No new API contract:** the dial calls the exact same `device.setpoint`
  tRPC mutation that the existing +/− buttons and slider already call,
  with the same input shape, clamp, and step.
- **No backward-compatibility risk:** there are no existing external API
  consumers, exports, or integrations touching the setpoint control path,
  so nothing downstream can break from a frontend-only change.
- The existing device modal's tabs (History/Automations), temperature
  chart, room assignment, and rename controls all continue working
  exactly as today — only the modal's open/close presentation and its
  setpoint control (for valve devices) change.
- No special deployment window or CI/CD change — same lint+typecheck+test
  gate every other change in this project goes through.

## Non-Goals

- No change to product type (web app) or user base/scale (small,
  2–5 person org) — this feature doesn't alter either.
- No hard deadline; after-hours-only work.
- Avoid: extending the dial or shared-layout morph to other controls —
  no dial-ification of other inputs (e.g. humidity thresholds,
  automation schedule times) and no morph animation for other UI
  surfaces (room cards, settings cards). Scoped strictly to the device
  card/modal setpoint control and its open/close transition.
- Avoid: a full design-system animation overhaul — other existing
  transitions (toasts, dropdowns, dialogs elsewhere in the app) are
  untouched by this change.

## Forward: tech-stack

The following stack-shaped decisions came up during shaping but are
explicitly NOT decided here — they belong to downstream tech-stack
selection / implementation planning:

- **Animation library choice:** Framer Motion vs. the native View
  Transitions API for the shared-layout card→modal morph. Each has
  tradeoffs (library weight + broad browser support vs. zero-dependency
  + newer/narrower browser support) — left open for the implementer.
- **Dial rendering approach:** SVG vs. Canvas for the thermostat dial.
  SVG is more idiomatic for a React app and easier to style/animate
  declaratively; Canvas may offer smoother drag performance at the cost
  of imperative rendering code. Left open.
- **Gesture handling library:** whether to hand-roll pointer-event-based
  angle math, or use a gesture library (e.g. `@use-gesture/react`,
  `framer-motion`'s built-in drag). Left open.
- **Reduced-motion implementation:** likely the CSS
  `prefers-reduced-motion` media query or its JS equivalent
  (`window.matchMedia`) — implementation detail for downstream.

## User Stories

### US-01: Admin adjusts a valve's setpoint via the new thermostat dial, with a morphing modal

- **Given** an admin is viewing the dashboard with at least one valve
  device card
- **When** they click the device card, watch it morph into the full
  modal, then drag the circular dial to a new setpoint
- **Then** the background shifts color smoothly as they drag, the same
  setpoint mutation fires as today (same 5–35°C clamp, 0.5°C step), and
  closing the modal reverses the morph back into the card

#### Acceptance Criteria
- The dial-set value matches exactly what the old +/− buttons / slider
  would have sent for the same gesture endpoint
- Sensor/plug device cards and their modals are visually and
  functionally unaffected except for gaining the shared-layout morph
- The modal's History/Automations tabs and temperature chart still work
  unchanged after the morph completes
- A failed setpoint mutation (network error) surfaces the same error
  toast behavior as today, not a silently stuck dial
